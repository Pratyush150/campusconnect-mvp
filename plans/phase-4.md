# Sprint 4 — Mentorship

**Goal:** Mentor-side product, orthogonal to assignment flow. Any client or doer can book a paid slot.

## Scope

- Admin invite flow: `POST /api/admin/mentors/invite` generates a single-use token; email contains `/register/mentor?token=...`
- Mentor onboarding form → `MentorProfile` row with `isApproved=false`
- Admin approves: `PUT /api/admin/mentors/:id/approve`
- Public listing: `GET /api/mentors` (anyone logged in, any role except anonymous — matches spec §9.1)
- Slot management:
  - `POST /api/mentor/slots` — accepts array; DB unique `(mentorId, slotDate, startTime)`
  - Recurring slots: `isRecurring=true, recurrenceDay=0..6` — generated by `/api/admin/cron/generate-recurring-slots`
- Booking:
  - `POST /api/mentors/:id/book { slotId, topic }` → creates Payment (mock), holds slot (pessimistic: `isBooked=true` inside a transaction)
  - On `captured`, booking is `confirmed`, meeting link attached (dev: placeholder; prod: Google Meet API)
  - Cancellation window: `mentor_cancellation_window_hours` setting (default 4h) — outside window = full refund, inside = 50% charge
- Subscriptions: **stubbed**. Row is created on purchase, no auto-renewal cron (deferred to Sprint 5).
- Session completion: mentor calls `PUT /api/mentor/bookings/:id/complete`; cron auto-completes after end-time+30m
- Rating: student rates 1–5 post-session

## Anti-leakage (L4)

- Meeting link is platform-generated only (dev: random room id URL)
- Mentor agreement text displayed at onboarding includes non-solicitation clause
- Report form on every booking for off-platform-contact attempts

## Definition of done

A client can browse mentors, book a slot, mock-pay, see confirmation, and mentor sees the booking on their calendar.
